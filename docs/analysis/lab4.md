# Лабораторна 4: Синхронна та асинхронна комунікація

## Що змінилося порівняно з лабораторною 3

У Lab 3 вся логіка виконувалась в лінію: Controller => Handler => Domain => Repository. Побічних
операцій не існувало — Handler виконував основну операцію і повертав результат.

У Lab 4 з'явився допоміжний компонент — `NotificationService`, який реагує на бізнес-дії
(створення та скасування бронювання). Handler тепер не тільки виконує основну операцію, а й
ініціює побічну — через один із двох механізмів комунікації.

Структурно додались:
- `application/interfaces/notification-service.ts` — контракт побічного компонента
- `application/interfaces/event-bus.ts` — абстракція для публікації подій
- `domain/events/integration-events.ts` — визначення подій (BookingCreated, BookingCancelled)
- `infrastructure/notifications/` — реалізація NotificationService
- `infrastructure/messaging/` — KafkaEventBus та NotificationSubscriber

## Побічні операції

Аналіз юзкейсів системи виявив операції, які мають відбуватися у відповідь на бізнес-дії,
але не є частиною основної логіки:

| Бізнес-дія | Побічна операція | Обґрунтування |
|---|---|---|
| Створення бронювання | Надіслати підтвердження | Клієнт хоче знати, що бронювання успішне. Але факт бронювання не залежить від того, чи email дійшов |
| Скасування бронювання | Надіслати повідомлення про скасування | Скасування має відбутися навіть якщо email не дійшов |

**Чому саме нотифікація?** Це природна побічна операція для системи бронювання квитків —
клієнт очікує зворотного зв'язку, але основна бізнес-операція не повинна залежати від
доступності каналу доставки повідомлень.

## Синхронна комунікація

### Як працює

Handler після основної операції **напряму викликає** `NotificationService` через інтерфейс:

```
CreateBookingHandler
  ├── bookingFactory.create()   ← основна операція
  ├── bookingRepo.save()        ← збереження
  └── notificationService.sendBookingConfirmation()  ← побічна (sync)
```

### Обробка помилок

Рішення: помилка нотифікації ігнорується (try/catch). Бронювання вже збережене в БД —
відкочувати його через збій email не має сенсу.

```typescript
try {
  await this.notificationService.sendBookingConfirmation(data);
} catch (err) {
  console.error('[SYNC] Notification failed, booking saved:', err);
}
```

### Спостереження

- **Час відповіді API:** сума часу основної операції + час нотифікації
- **Зв'язаність:** Handler знає про `NotificationService` (залежить від інтерфейсу)
- **При збої:** основна операція завершується успішно, нотифікація пропускається

## Асинхронна комунікація (Kafka)

### Як працює

Handler після основної операції публікує подію в Kafka topic. `NotificationSubscriber`
споживає повідомлення з Kafka і делегує `NotificationService`.

```
CreateBookingHandler
  ├── bookingFactory.create()   ← основна операція
  ├── bookingRepo.save()        ← збереження
  └── eventBus.publish(BookingCreated)  ← побічна (async)
       ↓ (Kafka topic: booking-events)
  NotificationSubscriber.onBookingCreated()
       └── notificationService.sendBookingConfirmation()
```

Handler не чекає поки підписник обробить подію. Повертає `{ id }` одразу після `publish()`.

### Вимоги до подій

- **Минулий час:** `BookingCreated` (не `CreateBooking`) — описує факт, що вже відбувся
- **Незмінні (immutable):** `readonly` поля, подія не модифікується після створення
- **Достатній контекст:** підписник не робить додаткових запитів до БД

### Ідемпотентність

Kafka гарантує at-least-once delivery — повідомлення може бути доставлене більше одного разу.
Для нотифікацій дублікат (повторний email) не є критичним. В production-системі додали б
deduplication за `bookingId` перед відправкою.

## Порівняння підходів

| Критерій | Синхронна | Асинхронна (Kafka) |
|---|---|---|
| **Час відповіді** | Основна операція + нотифікація | Тільки основна операція |
| **Зв'язаність** | Handler → NotificationService (знає інтерфейс) | Handler → EventBus (не знає про підписників) |
| **При збої нотифікації** | Помилка ловиться try/catch, основна операція успішна | Kafka retry, dead letter queue. Основна операція не залежить |
| **Консистентність** | Негайна (strong) — все в одному потоці | Відкладена (eventual) — підписник обробляє пізніше |
| **Складність** | Низька — прямий виклик через інтерфейс | Вища — брокер, серіалізація, consumer group |
| **Масштабування** | Обмежене — один потік | Consumer group масштабується горизонтально |
| **Додавання підписників** | Потребує зміни Handler | Новий subscriber, Handler не змінюється |
| **Відлагодження** | Простіше — один потік виконання | Складніше — розподілений потік, окремі логи |

## Який підхід для production?

Для нотифікацій (email, SMS, push) — асинхронний підхід оптимальний:
- Нотифікація може зачекати секунду — eventual consistency прийнятна
- Клієнт не повинен чекати поки email відправиться
- Якщо email-сервіс недоступний — Kafka зберігає повідомлення, retry відбудеться автоматично
- Можна додати нових підписників (аналітика, аудит) без зміни Handler

Для критичних операцій (наприклад, оплата) — синхронний: без оплати бронювання не може
бути підтверджене, потрібна негайна відповідь.

## Переключення режимів

Режим визначається змінною середовища `NOTIFICATION_MODE` в `.env`:

```env
NOTIFICATION_MODE=sync    # прямий виклик NotificationService
NOTIFICATION_MODE=async   # публікація подій через Kafka
```

Зміна значення + перезапуск сервера переключає всю систему між режимами без зміни коду.
