# Фінальний Аналіз та Ретроспектива (Лабораторна робота №5)

## 1. Аналіз Лабораторної 5: Modular Monolith

### 1.1 Як ви визначили межі контекстів? Які альтернативи розглядали?
Межі контекстів (Bounded Contexts) були визначені на основі бізнес-можливостей та аналізу життєвого циклу даних. Ми виділили три основні модулі:
1. **Catalog (Каталог)** — Управління інфраструктурою (станції, маршрути, потяги, вагони, розклад). Це административний контекст.
2. **Booking (Бронювання)** — Робота з клієнтами (реєстрація, авторизація) та транзакційна логіка бронювання місць.
3. **Analytics (Аналітика)** — Асинхронний збір статистики продажів.

**Альтернатива, яку ми розглядали:**
Виділення Auth у незалежний модуль: Розглядалося, але оскільки користувачі наразі існують виключно в контексті бронювання квитків, ми залишили реєстрацію в модулі Booking для зменшення накладних витрат.

### 1.2 Як ACL захищає модулі від змін в інших модулях?
**Anti-Corruption Layer (ACL)** виступає щитом між внутрішньою моделлю модуля та зовнішніми даними. У нашому проєкті ACL реалізовано у вигляді `CatalogAdapter` всередині модуля Booking.

**Механізм захисту:**
1. Модуль Catalog експортує публічний `CatalogFacade` та DTO `TrainBasicInfo`.
2. Модуль Booking має свій власний доменний інтерфейс `CatalogService` та доменну модель `TrainInfo`.
3. `CatalogAdapter` реалізує `CatalogService`, викликає `CatalogFacade`, та транслює зовнішній DTO у внутрішню модель.
*Результат*: Якщо модуль Catalog вирішить повністю змінити структуру бази даних, ORM, або формат `TrainBasicInfo`, модуль Booking не зазнає жодних змін у своїй бізнес-логіці. Нам доведеться оновити лише один файл — `CatalogAdapter`, який візьме на себе удар.

### 1.3 Де у системі strong consistency, а де eventual consistency? Чому це прийнятно?
- **Strong Consistency**: Використовується під час створення бронювання. Модуль Booking робить синхронний виклик до `CatalogFacade`, щоб перевірити, чи дійсно існує потяг і чи вільне конкретне місце (`verifySeatExists`). Це критично необхідно, адже ми не можемо продати квиток на неіснуюче місце (Double Booking). Тут дані повинні бути узгоджені "в цю ж мілісекунду".
- **Eventual Consistency**: Використовується для модуля Аналітики. Коли бронювання створено, Booking публікує асинхронну подію `BookingCreated` у **Kafka**. Модуль Analytics через `AnalyticsSubscriber` отримує її з затримкою і оновлює статистику. Це абсолютно прийнятно, оскільки затримка оновлення графіків продажів на пару секунд не порушує жодних критичних бізнес-правил.

### 1.4 Що зміниться, якщо потрібно буде виділити один з модулів у окремий сервіс?
Завдяки правильно побудованому Modular Monolith, виділення модуля Analytics або Catalog у мікросервіс буде надзвичайно простим:
1. **Бізнес-логіка (Domain/Application) залишиться незмінною.**
2. У модулі Booking `CatalogAdapter` перестане викликати `CatalogFacade` in-memory, а замість цього почне робити HTTP (REST) або gRPC запит до нового мікросервісу Catalog.
3. Модуль Analytics вже отримує дані через Kafka, тому його достатньо просто запустити в іншому Docker-контейнері з іншою базою даних. Він продовжить слухати той самий топік Kafka без жодних змін у коді.

## 2. Ретроспектива курсу (від Лаби 1 до Лаби 5)

### 2.1 Як змінювалася архітектура проєкту на кожному етапі?
- **Лаба 1-2**: Ми перейшли від простого CRUD (де контролери робили прямі запити до БД) до **Layered Architecture**. Були виділені шари Presentation, Application (Use Cases), Domain та Infrastructure. З'явилася сувора ізоляція обов'язків.
- **Лаба 3-4**: Впровадження **Rich Domain Model** та **CQS/CQRS**. Ми прибрали логіку з сервісів (Anemic Model) і перемістили її в сутності (`Train`, `Booking`). Для читання даних (GET-запити) ми відокремили Read-репозиторії від Write-репозиторіїв. З'явився внутрішній Event Bus для сповіщень.
- **Лаба 5**: Трансформація у **Modular Monolith**. Розділення проєкту на незалежні бізнес-модулі (`booking`, `catalog`, `analytics`), впровадження Kafka для асинхронної взаємодії, створення Facades та ACL для публічних контрактів.

### 2.2 Які архітектурні рішення виявилися найціннішими?
1. **Dependency Injection (DI)**: Передача репозиторіїв через конструктори у `server.ts` дозволила легко писати Unit-тести (передаючи Mock-об'єкти) і гнучко замінювати реалізації.
2. **Command-Query Separation (CQS)**: Відокремлення логіки запису (команди, які міняють стан і використовують Domain Model) від логіки читання (запити, які швидко тягнуть DTO з бази). Це значно пришвидшило роботу API.
3. **Event-Driven Architecture (через Kafka)**: Дозволила повністю відв'язати другорядну логіку (статистику, сповіщення) від основного транзакційного потоку.

### 2.3 Що було б зроблено інакше, знаючи кінцевий результат?
1. **Планування меж модулів із самого початку**: На початку розробки ми мали спільні папки `domain`, `application` для всіх сутностей. Через це під час переходу до Modular Monolith довелося робити масштабний рефакторинг і переписувати сотні імпортів. Якби ми почали з модульної структури одразу, ми б заощадили багато часу.
2. **Read-моделі (Prisma Views)**: Замість використання standard Prisma Client для складних запитів читання (наприклад, пошук вільних місць), можна було б одразу спроєктувати спеціальні SQL запити або Read Models для забезпечення кращої продуктивності CQRS.

### 2.4 Які trade-offs (компроміси) ви побачили між простотою та гнучкістю?
Побудова системи за принципами DDD та Modular Monolith вимагає значних інвестицій часу:
- **Негатив (Складність)**: Замість того, щоб написати 1 файл контролера на 50 рядків, який робить все, нам доводиться створювати Controller, Command Handler, Domain Entity, Factory, DTO, Repository Interface та Prisma Repository Implementation. Кількість коду зростає в рази (Boilerplate).
- **Позитив (Гнучкість)**: Ця складність повністю окупається, коли бізнес-логіка стає складною. Якщо завтра зміниться база даних, ORM, або з'явиться вимога зробити мікросервіси — наша система готова до цього майже без змін у бізнес-логіці. Код легко тестувати (понад 112 тестів), легко масштабувати в команді (кожен розробник може безпечно працювати над своїм модулем).

**Підсумок**: DDD та Modular Monolith — це інструмент не для швидких MVP, а для складних Enterprise-проєктів, які розраховані на довгу підтримку та розвиток. Для даної предметної області (система бронювання залізничних квитків) цей підхід повністю себе виправдав.
